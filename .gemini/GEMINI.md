# Gemini Project Guide: Kkookk Digital Stamp Service

## 1. 프로젝트 개요

이 프로젝트는 'KKOOKK' 디지털 스탬프 SaaS의 백엔드와 프론트엔드를 개발합니다.

- **`backend/`**: Spring Boot API (Java 17, MySQL)
- **`frontend/`**: React + TypeScript 웹 앱
- **주요 사용자 흐름**:
    1.  **스탬프 발급**: 고객의 요청을 단말기에서 승인합니다.
    2.  **보상 사용**: 고객이 OTP 인증 후 보상을 사용하면, 점주가 이를 확인합니다.
    3.  **마이그레이션**: 종이 스탬프 카드를 사진으로 등록하면, 점주가 수동으로 승인하여 디지털 스탬프를 적립해줍니다.

## 2. 저장소 구조

핵심 코드는 `backend`와 `frontend` 디렉토리에 있으며, 각 디렉토리에는 더 상세한 `GEMINI.md` 가이드가 있습니다. 저를 위한 상세한 기술 지시사항과 가이드는 `.gemini/` 폴더에 있습니다.

```
/
├─ .gemini/         # 저(Gemini)를 위한 스킬 및 가이드
├─ backend/
│  └─ GEMINI.md    # 백엔드 전용 가이드
├─ frontend/
│  └─ GEMINI.md    # 프론트엔드 전용 가이드
└─ GEMINI.md        # 최상위 프로젝트 가이드 (이 파일)
```

## 3. 나의 작업 원칙: 설계 → 제안 → 구현 → 검증

**코드를 바로 작성하지 않습니다.** 사용자의 모든 요청에 대해 다음 단계를 따릅니다.

1.  **요구사항 분석 및 설계**: 주어진 요구사항을 분석하고, 1-2개의 설계 옵션을 고려합니다.
2.  **계획 제안 및 승인**: 가장 합리적인 설계와 구현 계획을 사용자에게 제안하고, **반드시 승인**을 받습니다.
3.  **파일 단위 구현**: 승인된 계획에 따라 파일 단위로 코드를 변경, 생성합니다.
4.  **검증**: 코드 변경 후에는 반드시 린트, 테스트, 빌드를 실행하여 코드의 정합성과 품질을 확인합니다.

## 4. 작업 완료 조건 (Definition of Done)

다음 체크리스트를 통과해야 제 작업이 완료된 것으로 간주합니다.

- [ ] 린트(Lint) 실행 통과
- [ ] 테스트(Test) 실행 통과
- [ ] 변경된 파일에 `TODO` 주석이 없음
- [ ] 예상되는 실패 케이스에 대한 에러 핸들링 구현
- [ ] API 변경 시 DTO와 상태 코드를 명확히 문서화
- [ ] 관련된 README나 문서 파일 업데이트
