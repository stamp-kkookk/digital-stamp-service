## 1. 커밋 메시지 구조

기본적인 커밋 메시지는 제목(subject)과 본문(body)으로 구성하며, 각 파트는 빈 줄을 두어 구분합니다.

```java
type: Subject  // 제목
<empty line>
body           // 본문 (선택 사항)
```

- **제목(Subject):** `type: Subject` 형태를 사용하며, `:` 뒤에 공백(space)을 추가합니다.
- **본문(Body):** 긴 설명이 필요한 경우 선택적으로 작성합니다



---

## 2. 타입(Type)

어떤 의도로 커밋을 했는지 명시하며 영어로 작성하되 첫 문자는 대문자로 시작합니다.

| **타입(Type)** | **설명** |
| --- | --- |
| **feat** | 새로운 기능을 추가한 경우  |
| **fix** | 버그를 수정한 경우  |
| **refactor** | 리팩토링(비즈니스 로직 변경 없음)  |
| **style** | 코드 형식, 세미콜론 추가 등 스타일 변경  |
| **docs** | 문서 수정(README, 주석 추가/변경 등)  |
| **test** | 테스트 코드 추가, 수정, 삭제  |
| **chore** | 기타 변경사항(빌드 스크립트, 패키지 매니저 등)  |
| **init** | 프로젝트 초기 생성  |
| **rename** | 파일 혹은 폴더명을 수정하거나 옮기는 경우  |
| **remove** | 파일을 삭제하는 작업만 수행한 경우  |

---

## 3. 제목(Subject) 규칙

코드 변경 사항에 대한 짧은 요약을 의미하며 다음 규칙을 준수합니다.

- **길이 제한:** 최대 50자를 넘기지 않습니다.
- **기호 사용:** 마침표(`.`)나 특수기호를 끝에 붙이지 않습니다.
- **한글 작성 시:** "추가", "고침", "변경" 등 명령조의 개조식 구문으로 작성합니다.

---

## 4. 본문(Body) 규칙

부연 설명이 필요하거나 커밋의 이유를 설명할 때 선택적으로 작성합니다

- **내용 구성:** '어떻게' 변경했는지보다 **'무엇을'**, **'왜'** 변경했는지를 설명합니다.
- **구분:** 제목과 본문 사이에 반드시 빈 줄 한 줄을 추가합니다.
- **길이:** 한 줄당 최대 75자(또는 72자)를 넘기지 않도록 합니다.
- **리스트 활용:** 여러 줄의 메시지를 작성할 때는 `-`로 구분하여 나열할 수 있습니다.
